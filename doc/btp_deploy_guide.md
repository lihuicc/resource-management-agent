> 本章记录在 SAP AI Launchpad 中完成的所有配置步骤。

### 2.1 访问 AI Launchpad

登录 BTP Cockpit，从 Service Marketplace 或 Instances and Subscriptions 中打开 **SAP AI Launchpad**。

### 2.2 创建 Workspace

1. 进入 **Workspaces** → **Add**
2. 填写：
   - **Name**: `resource-management-agent`
   - **AI Core Connection**: 选择已有的 AI Core 服务实例
3. **Save**

### 2.3 Resource Group

使用默认 Resource Group：`default`

### 2.4 创建 Claude 模型 Configuration

1. 进入 Workspace → **ML Operations** → **Configurations**
2. 检查是否已有 Claude 模型的 Configuration
3. 如果没有，点击 **Create**，填写：
   - **Name**: `anthropic--claude-4.5-sonnet_autogenerated`
   - **Scenario**: `foundation-models`
   - **Executable**: `aws-bedrock`
   - Version: 选择最新版本
4. 展开 **Parameters**，填写：
   - **modelVersion**: `latest` (Default)
   - **modelName**: `anthropic--claude-4.5-sonnet`
5. **Save**

1. 进入 Workspace → **ML Operations** → **Deployments** → **Create**
2. 填写：
   - **Configuration**: 选择 `anthropic--claude-4.5-sonnet_autogenerated`
3. **Create**
4. 等待 Status 变为 **Running**（约 2-5 分钟）
5. 记录 **Deployment ID**（格式如 `d1234567890abcdef`），后续环境变量 `AICORE_DEPLOYMENT_ID` 会用到
   - Deployment URL: `https://api.ai.intprod-eu12.eu-central-1.aws.ml.hana.ondemand.com/v2/inference/deployments/dc8d78435d006caf`

## 3. BTP 基础服务配置

### 3.1 前置条件确认

在开始之前，确认以下资源已在 BTP Subaccount 中就绪：

### 3.2 创建 HANA Cloud 实例并绑定 Space

1. 登录 **BTP Cockpit** → 你的 Subaccount
2. 进入左侧菜单 **SAP HANA Cloud**
3. 检查是否已有实例：
   - **已有实例**：点击实例 → **CF Mappings** → **Edit** → 添加 org `CN_SDC_Tenant_cn-sdc-subaccount-virtualproj-59qj1osn` 和 space `dev-resource-management-agent` → **Save**
   - **没有实例**：点击 **Create** → **SAP HANA Cloud, SAP HANA Database**，填写：
     - Instance Name: 随意，如 `rm-agent-hana`
     - Administrator Password: 设置并记录
     - 规格选 Free Tier 或最小规格
     - **CF Mappings**：选择 org 和 space `dev-resource-management-agent`
     - 点击 **Create**，等待约 10-15 分钟直到状态变为 **Running**

> **注意：** HANA Cloud 实例名只是标识符，随意起。真正重要的是 CF Mappings 绑定到正确的 Space，否则 HDI Container 无法创建。

## 4. 代码改动

### 4.1 agent-service.js — AI Core 认证适配（必须）

测试已确认 AI Core (`aws-bedrock` executable) 接受 **Anthropic 原生格式**请求和响应，tool-use 循环逻辑无需改动。

唯一需要适配的是认证方式：

| | 当前（本地） | BTP 生产 |
|---|---|---|
| 认证 header | `x-api-key: <ANTHROPIC_API_KEY>` | `Authorization: Bearer <OAuth token>` |
| 额外 header | 无 | `AI-Resource-Group: default` |
| Token 来源 | 固定 API Key | 从 AI Core VCAP_SERVICES 的 clientid/clientsecret 换取 |

改动方案：在 BTP 环境下（`VCAP_SERVICES` 存在时），自动用 client credentials flow 获取 Bearer token，并通过 `@anthropic-ai/sdk` 的 `defaultHeaders` 注入。由于顶层不能直接 `await`，token 获取做成懒加载（首次调用时初始化）：

```js
// srv/agent-service.js 顶部初始化逻辑（伪代码）
const isAICore = !!process.env.VCAP_SERVICES;

let _client = null;

async function getClient() {
  if (_client) return _client;

  let headers = {};
  if (isAICore) {
    const creds = JSON.parse(process.env.VCAP_SERVICES).aicore[0].credentials;
    const res = await fetch(`${creds.url}/oauth/token`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
      body: `grant_type=client_credentials&client_id=${encodeURIComponent(creds.clientid)}&client_secret=${encodeURIComponent(creds.clientsecret)}`
    });
    const token = (await res.json()).access_token;
    headers = {
      'Authorization': `Bearer ${token}`,
      'AI-Resource-Group': process.env.AICORE_RESOURCE_GROUP || 'default'
    };
  }

  _client = new Anthropic({
    apiKey: isAICore ? 'dummy' : process.env.ANTHROPIC_API_KEY,
    baseURL: process.env.ANTHROPIC_BASE_URL,
    defaultHeaders: headers
  });
  return _client;
}

// 在 runAgent 中用 getClient() 替代直接引用 client
```

> **注意：** OAuth token 有效期约 12 小时，需要在到期前刷新。生产代码需加 token 缓存和自动刷新逻辑（记录 token 获取时间，超过 11 小时则重新获取）。

### 上传 502 Bad Gateway / mtar 包过大
```
Could not upload file: 502 Bad Gateway
```
mtar 包过大（数 GB）导致上传超时。根本原因是 `mta.yaml` 中 srv 模块的 `path` 指向了项目根目录 `.`，导致 node_modules 等被打包进去。

解决：将 srv 模块的 `path` 改为 `gen/srv`，并在 `before-all` 中加入 `cds build --production`，让 mbt 只打包编译产物。参考 `mta.yaml` 中当前配置。

### Cannot find '@sap/xssec'

```
Error: Cannot find '@sap/xssec'. Make sure to install it with 'npm i @sap/xssec'
```

CDS 在 production 模式下启用 xsuaa 认证时需要 `@sap/xssec`，但该包未在 `package.json` 的 `dependencies` 中声明（注意必须是 `dependencies`，不能是 `devDependencies`，否则 BTP 上不会安装）。

解决：在 `package.json` 的 `dependencies` 中加入 `"@sap/xssec": "^4"`，重新 `mbt build` 后 deploy。

### welcomeFile 404 — CAP 去掉 app/ 前缀

App Router 登录后跳转到 `welcomeFile` 返回 404。原因是 CAP 在 serve 静态文件时会去掉 `app/` 前缀：

| 源码路径 | CAP 实际 serve 路径 |
|---------|-------------------|
| `app/resource-ui/webapp/index.html` | `/resource-ui/webapp/index.html` |

解决：`approuter/xs-app.json` 的 `welcomeFile` 设为 `/resource-ui/webapp/index.html`，不加 `app/` 前缀。

### cp 命令在 Windows 不可用

`mta.yaml` 的 `before-all` 中使用 `cp -r` 在 Windows 环境下会报错：
```
exec: "cp": executable file not found in %PATH%
```

解决：改用 node 内联脚本替代，Windows/Linux 均兼容：
```yaml
- node -e "const fs=require('fs');const path=require('path');function cp(src,dst){fs.mkdirSync(dst,{recursive:true});fs.readdirSync(src).forEach(f=>{const s=path.join(src,f),d=path.join(dst,f);fs.statSync(s).isDirectory()?cp(s,d):fs.copyFileSync(s,d);})}cp('app','gen/srv/app')"
```
